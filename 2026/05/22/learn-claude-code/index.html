Chapter 1:Agent = LLM + TOOL + LOOP

1. 核心思想

Agent就是一个循环：

LLM 本身无状态，每次调用都不知道之前说过什么，所以必须把完整的对话历史（包括工具调用和结果） 一次次传回去，它才能”记得”上下文。这是 agent loop 存在的根本原因。

2. 工具定义——Bash

只提供一个工具bash，作用是开一个子进程shell来执行命令：

  • shell=True：通过系统 shell（Windows 的 cmd.exe / Linux 的 sh）执行，支持管道、重定向
  • capture_output=True：捕获输出，不打印到终端
  • text=True：以字符串（而非 bytes）形式返回
  • timeout=120：超时 120 秒，超时抛异常
  • 如果任务执行失败，返回相关信息给LLM
  • subprocess.run()：阻塞等待

3. 三层关系：用户 ↔ Client ↔ LLM

整个系统涉及三个角色：

  • 用户：输入文本指令，读取最终回复
  • Client：Anthropic() 创建的客户端对象，负责组装消息、通过 HTTP 发送请求、解析响应。它是一个本地代理，不包含任何模型权重
  • LLM：运行在远程服务器上的大模型，Client 通过 API 调用它，每次调用都是独立的

Client可以通过 base_url 指向不同的提供商——Anthropic 官方、DeepSeek 兼容接口等，LLM 在远端，Client 的代码不用动。

4. 工具注册

所有工具统一注册进 TOOLS 列表，格式由 Anthropic API 规定：

  • input_schema 是 JSON Schema 格式，告诉 LLM 这个工具有什么参数
  • required：调用时必须传的参数

Chapter 4：Subagent

1. 核心思想

每轮对话、读文件、跑命令的输出都保存进messages数组中，永久留在上下文里。因此Agent工作越久，上下文越臃肿。引入subagent专门用来执行任务，用独立的message[]，只将最终结果传给parent agent，任务执行后subagent的上下文就销毁，隔离上下文，不污染主对话。

2. 工具注册

subagent可以调用各种基础工具，但分配任务的task工具只有parent agent可使用，为了防止递归。

3. subagent执行逻辑

subagent 以 messages=[]完全空白的上下文启动, 运行自己的循环。只将最终文本返回给父 Agent。

对subagent循环轮次进行安全性限制。subagent可能跑了多次工具调用, 但整个消息历史直接丢弃。parent agent收到的只是一段摘要文本, 作为普通 tool_result 返回。实现父子上下文隔离。

提示parent agent可以分配任务

5. System Prompt For Subagent

提示subagent，对任务执行情况进行总结返回给parent agent。

6. 关键要点

subagent在自己的循环了做了什么事情、执行什么任务，外层完全看不见，只有任务执行结束才向返回其任务执行摘要summary，父子上下文完全隔离。

Chapter 6：Context Compact

1. 核心问题

多轮对话上下文太大，通过压缩上下文换来无限对话。

相关常量定义：

  • 第一层（micro_compact）：每轮调用 LLM 前，将旧的 tool_result 内容替换为 [Previous: used {tool_name}]，保留最近 KEEP_RECENT 条和 PRESERVE_RESULT_TOOLS 中的内容。
  • 第二层（auto_compact）：第一层压缩后 token 仍超过 THRESHOLD，则自动调用 LLM 生成摘要，替换整个 messages。
  • 第三层（manual compact）：token 未超阈值但 LLM 自己觉得上下文混乱时，主动调用 compact 工具请求压缩。

2. 第一层实现

micro_compact: 每次 LLM 调用前, 将旧的 tool result 替换为占位符

记录每一次工具调用的tool_use_id、output等相关信息

KEEP_RECENT:最近几次工具调用结果要保留，让LLM能够获得最新信息。

在PRESERVE_RESULT_TOOLS中的结果要保留，比如read_file。

这里的result是引用，messages会同步修改。messages结构完全不变，只是内容减少。

3. 第二层实现

auto_compact: 经过第一层压缩后，token仍然超过阈值时, 保存完整对话到磁盘, 让 LLM 做摘要。

将完整对话内容写进磁盘，每次写进的文件用时间戳命名，保证唯一性。

通知LLM压缩对话，prompt要求总结对话要保持连贯性，必须包括的内容有：已完成的操作、当前的状态、关键决策，简洁的同时保留关键细节。

但有一个缺陷：json.dumps(messages, default=str)[-80000:] 只取了序列化后的最后 80000 个字符。对话的开头部分（用户的原始指令、任务目标）在序列化字符串的前面，最容易被截掉。而对话的末尾都是工具执行结果和中间输出。结果 LLM 做摘要时看不到原始任务目标，只能靠猜——这会导致幻觉，LLM 编造一个它认为合理的任务背景。这个实现是一个已知的简化，s_full.py 也沿用了同样的问题。

4. 第三层压缩

只有token数量大才会触发第二层压缩，但是LLM也可以自主决定进行内容压缩，比如当上下文比较混乱或结论不明确的时候，通过compact工具调用实现。

当LLM需要调用compact时，等到本轮所有操作执行完且执行结果都加入上下文后，才压缩上下文，然后直接结束循环，等待用户下一次输入。

Chapter 8：Background Task

1. 核心思想

run_bash是创建一个子进程跑命令，subprocess.run本身是同步阻塞的——它在哪个线程调用就阻塞哪个线程。如果操作耗时长，比如npm install、pytest、docker build、git push，整个循环需要长时间等待。如果不是立即需要结果，可以将其放在后台线程运行，待其完成后再通知LLM。

但是像run_write(),run_edit(),run_read()等操作执行速度快，立即需要执行结果，就不适合放在后台。

2. 解决方案

同时提供bash(blocking)和background_run(立即返回)两个工具，LLM根据描述选择。background_run会创建一个新线程，thread.start()后立即返回，新线程在后台继续运行subprocess.run()，运行结束后将结果写入队列self._notification_queue。harness在每次调LLM前通过drain_notifications()检查是否有已完成的任务并取出，注入成<background-results>消息后再调LLM。

Chapter 9：Agent Teams

1. 核心思想

s04 的子 agent 是一次性的（spawn → 执行 → 销毁），用完即弃。s09 改为持久化的队友模型：队友完成任务后转为 IDLE 状态，可以多次分配任务，直到主进程退出才消失。

真正的团队协作需要三样东西:

  • 能跨多轮对话存活的持久 Agent
  • Agent身份和生命周期管理
  • Agent 之间的通信通道

2. 解决方案

主线程agent作为leader，将任务分配给team agent，每个team agent任务执行完不会立即撤销，而是转为IDLE状态等待下次分配，直到主循环结束所有守护线程team agent才会撤销。

3. agent 之间的通信

INBOX_DIR路径下存放各个agent收到的信件，每个信件中的每条信息都是JSON格式。

发送消息： 所有agent都可以给其他agent发消息，发送方为sender，接收方为to，将message以JSON格式append到接收方的收件箱中。

接受消息： 每个agent在每轮循环前都会读取自己的信箱，并将消息取出加入到自己的上下文中，然后清空信箱。

广播消息： leader agent可以向teammates广播消息，即给每一位teammate发送这条消息send()。

注意：s09 的 MessageBus 没有加锁。与 s08 的 _notification_queue（内存列表，append 和 clear 会冲突）不同，s09 的邮件写入是文件追加（open("a")），每次 write() 在系统层面是原子操作，不会出现在同一个文件里两条消息交错混行的问题。但严格来说并发写入文件仍有概率丢失数据，教程代码为了突出团队通信模式而省略了锁——生产环境应加文件锁或改用数据库。

4. agent 团队管理

team agent 的相关配置都写在self.config_path中，每条配置都是JSON格式。

加载团队信息、更新团队信息、获取某个team agent相关信息。

leader agent可以分配任务，如果这个成员已存在且空闲，可分配给它，如果存在但不空闲就会提示错误信息。如果成员还不存在，就创建该成员并写进配置信息。让成员在新的线程中执行任务，主循环不会阻塞。

team agent任务执行完或者循环轮次超出限制后，就会置为IDLE状态。

这部分是team agent可使用的工具，与leader相比，不能spawn，不能broadcast，不能list_teammates。

这意味着队友间无法自主发现彼此。队友知道给谁发消息，只能通过：

  1. Lead 在 prompt 中告知 — spawn("alice", "coder", "写代码，完成后发给 bob")，初始消息直接写了分工
  2. Lead 发消息告知 — send_message("alice", "你的代码发给 bob review")，队友 drain 收件箱后读到
  3. 从收件箱 from 字段反推 — bob 主动发消息给 alice，alice 看到 "from": "bob" 即可回复

这是 s09 的设计限制：队友是被管理者，没有团队全局视角，只能通过 Lead 的分配和消息被动发现其他队友。

8. 关键要点

  1. s04 Subagent vs s09 Teammate — s04 一次性、同步阻塞、不持久；s09 持久化、线程并行、有邮箱通信
  2. config.json 只存队友 — lead 的信息不在配置中，lead 是主进程本身而不是 spawn 出来的
  3. shutdown_request 等消息已声明但未实现 — s09 定义了 5 种消息类型，但只实现了 message 和 broadcast 的处理逻辑，其余留给 s10 Team Protocols
  4. 进程退出≠ agent_loop 结束 — 队友是 daemon 线程，只有 Python 进程退出时才被 OS 强制回收。agent_loop 只是主线程里的一个函数，结束后回到 REPL 等待用户输入，进程依然活着，队友线程继续运行
  5. lead 不知道队友何时完成 — 没有回调或推送，完全依赖队友主动 send_message 汇报或 lead 主动 list_teammates 查询
  6. LLM 通过 list_teammates 发现队友 — LLM 看不到内存里的线程列表，只能调 list_teammates 查看所有队友的名称、角色和状态。如果 LLM 直接 spawn 同名队友，spawn() 内部会去重检测，存在且 idle 则复用，不重复创建

9. 用户交互

用户可通过输入/team,/inbox来查看系统状态。这两个命令只在 agent_loop 结束后（即回到 REPL 等待输入时）才能输入。

它们能查到信息是因为：

  • /team 直接读磁盘上的 .team/config.json — 队友线程在执行中会实时更新其中的 status，agent_loop 结束后进程还在，队友 daemon 线程继续运行，config 也会持续更新
  • /inbox 直接读 .team/inbox/lead.jsonl — 队友线程可在任何时候往里写消息，不依赖 lead 的 loop 是否在运行

注意：/inbox 会 drain 清空文件，而 lead 的 agent_loop 每轮也会 drain 收件箱。如果队友在 loop 结束前发了消息，已被 loop drain 走；如果在 loop 结束后才发，/inbox 才能查到。

10. 运行时局限分析

下面是多次运行 s09 暴露的设计局限。

10.1 时序竞态：发消息和读信箱的先后问题

队友线程启动后的第一件事是调用 LLM。LLM 返回后，线程要么继续（tool_use）要么退出（end_turn）。但 lead 发送消息是在 spawn 之后，无法保证消息在队友第一次读 inbox 之前到达：

Alice 的 read_inbox 在 lead 的 send_message 之前执行，消息到的时候 Alice 的线程已经结束了。这个竞态在 s09 中没有处理——lead 发后不管，不确认队友是否已接收。

10.2 队友 idle 后不会自动醒来

队友的循环结构本质上是一次性任务执行：

一旦 LLM 决定 end_turn，循环 break，线程函数返回，队友永久休眠。之后即使有新消息写入 inbox，也没有任何代码再次检查。真正的空闲轮询需要这样的结构：

这个机制留给了 s11 Autonomous Agents。

10.3 LLM 决策差异决定队友活跃度

Alice 和 Bob 执行完全相同的 _teammate_loop 代码，但行为截然不同：

Alice Bob
首次 LLM 响应 end_turn tool_use
循环次数 1 次 可持续数十次
最终状态 idle（线程终止） working（线程活跃）
能否收到后续消息 否 能

根源不在代码而在 LLM 的自主决策。Alice 的 LLM 认为”收到任务，待命即可”所以直接 end_turn。Bob 的 LLM 认为”我是 tester，先探索环境”于是开始调工具。由于 daemon 线程不随 agent_loop 终止，如果 Bob 一直 tool_use，他会在 REPL 后台持续运行，输出与用户交互交错。

10.4 队友通过 read_file 绕开 inbox 隔离

s09 的邮箱通信设计依赖于所有 agent 通过 send_message / read_inbox API 访问信箱。但队友拥有 read_file 工具，可以直接读取收件箱文件：

Alice 的 read_inbox 会 drain 清空文件，但 Bob 用 read_file 读到的是原始内容，不会被 drain 影响。这暴露了一个问题：文件系统作为通信通道时，读写权限无法按 agent 隔离。每个队友都能读/写所有 .jsonl 文件，没有访问控制。真正的解决需要文件权限控制或让队友不暴露 read_file 给队友。

10.5 多线程 stdout 交错

队友是独立线程，它们和 lead 都通过 print() 输出到同一个终端。多线程并发 print() 会导致输出行被拆散：

print() 在 Python 中不是原子的——写多行文本时可能被线程调度打断。这是 s09 的工程简化：用文件做通信隔离但没有做输出隔离。

10.6 广播不改变队友行为

当 lead 调用 broadcast 时，消息被写入每个队友的 inbox。队友在下轮循环中 drain 到对话历史：

没有根据消息类型做分支逻辑。广播消息被当作普通文本追加，LLM 看到后可能继续调工具，也可能回复文本。没有 shutdown_request 那样的退出响应——消息类型虽然定义了 5 种，但处理时全部一视同仁。这意味着：

  • 广播不能”打断”正在工作的队友
  • 广播不能改变队友的循环行为（让它退出、暂停等）
  • 即使收到 broadcast，LLM 是否响应完全取决于自主判断

10.7 Lead 空转问题

Lead 的任务分配是”发后不管”——spawn 和 send_message 都立即返回，不等待队友完成。之后 lead 进入循环反复检查：

lead 没有任何超时或重试机制。如果队友忙不过来或消息丢失，lead 只会无限空转直到 LLM 自己觉得”任务完成”为止。这个决策完全依赖 LLM 的判断，不是由代码保障的。

Chapter 10：Team Protocols

1. 核心思想

lead-teammate之间的通信如何协调

  • 关机：lead认为teammate不再需要了，可以要求其关机。但直接关闭会留下写了一半的文件和过期的 config.json。需要双方握手达成一致后才继续。
  • 计划审批：teammate执行任务前向lead报备，审核无误后再继续。

s10通信类型

2. 解决方案

s09 的消息系统是非结构化的——发什么内容、对方怎么响应全靠 LLM 自己理解。s10 在 harness 层加入：

  1. 消息类型区分 — shutdown_request、shutdown_response、plan_approval_response 新增三种新类型，LLM 看到类型就知道该怎么处理
  2. request_id 关联 — 每条请求带唯一 ID，响应引用同一 ID，保证请求和响应的对应关系
  3. 服务端追踪 — shutdown_requests 和 plan_requests 两个全局字典记录每条请求的生命周期：pending → approved/rejected
  4. Harness 层执行 — LLM 只做决策（approve true/false），真正让线程退出的是代码中的 should_exit 标记

3. 工作原理

两个字典分别记录关机请求和计划请求的生命周期。用 _tracker_lock 保护，因为 lead 和队友的 handler 可能在不同线程中读写同一个字典。

send() 方法支持 extra 参数，将结构化数据合并进消息：
没有 extra，消息只有 type/from/content/timestamp 四个固定字段。有 extra 就可以带 request_id、approve、plan 等结构化数据，接收方不需要从文本中解析。

Lead 发起关机（shutdown_request）:

  1. 生成 8 位唯一 ID req_id
  2. 在 shutdown_requests 中记录 {target: "teammate", status: "pending"}
  3. 发 shutdown_request 类型消息给 teammate，extra 中带 request_id
  4. 返回 req_id 给 lead LLM

队友响应关机（shutdown_response）,LLM决定approve or reject

  1. 提取 request_id 和 approve（都是必填参数，用 args["key"] 索引）
  2. 更新服务端追踪器的状态为 approved 或 rejected
  3. 发 shutdown_response 消息给 lead，extra 中带回 request_id 和 approve
  4. 可选参数 reason 用 args.get("reason", "") 读取，不存在时为空字符串

关键设计：LLM 只决定 approve 与否，harness 层负责执行退出。即使 LLM 批准了却继续调工具，should_exit 标记会在下一轮循环检查时直接 break。状态标记为 shutdown表示由lead强制关机，idle表示自行结束任务空闲。

队友提交计划（plan_approval）：

  1. teammate生成 request_id
  2. 记录到 plan_requests，包含提交者、计划全文、状态 pending
  3. 发 plan_approval_response 消息给 lead，extra 中带 request_id 和 plan
  4. 返回值包含 "Waiting for lead approval"，让teammate LLM 知道需要等待，不会立即执行变更。

Lead 审批计划（handle_plan_review）:

  1. 用 plan_requests.get(request_id) 查找请求
  2. 未找到直接返回错误字符串，LLM 可以纠正 request_id 后重试
  3. 找到后更新状态为 approved/rejected
  4. 发 plan_approval_response 类型消息给提交者，extra 中带 request_id、approve、feedback

Lead 查看关机状态（check_shutdown_status）:

lead 发完 shutdown_request 后可查询队友是否已响应。

8. 关键要点

  1. request_id 模式 — 两个协议共用一个设计：唯一 ID 关联请求和响应。无论中间有多少其他消息混入，响应方的 request_id 能准确对应原始请求

  2. extra 字段传递结构化数据 — s09 的 content 像邮件正文，自由文本。s10 的 extra 像结构化附件，机器可解析，用于协议的正常运行（关联请求和响应、传递审批状态）

  3. 两种消息类型 vs 一种 — Shutdown 用了 shutdown_request + shutdown_response 两种类型，因为 harness 层需要检查 block.name == "shutdown_response" 来触发 should_exit。Plan approval 只用了 plan_approval_response 一种类型，因为 harness 层不需要做特殊处理——消息类型是给 LLM 看的，靠 extra 字段区分”提交”还是”审批结果”

  4. Lead 和队友的同名工具含义不同 — 队友的 shutdown_response 是响应关机（传 approve），lead 的 check_shutdown_status 是查询状态（传 request_id，返回 pending/approved/rejected）。两个工具功能完全相反

  5. shutdown 不释放资源 — shutdown 只改 config.json 中的状态为 shutdown，不删除 entry、不释放线程（线程早已自己结束）。唯一作用是允许带新 role 重新 spawn——spawn() 检查 member["status"] in ("idle", "shutdown") 才允许复用

  6. 队友间通信仍靠 lead 介绍 — s10 没有给队友加 list_teammates，队友知道对方的唯一途径仍是 lead 在 prompt 或消息中告知。协议能力增强了，团队拓扑仍是 lead-centric
